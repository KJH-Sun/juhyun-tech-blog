대규모 마이크로서비스 코드베이스를 다루다 보면, 답을 찾는 일보다 "어디서부터 찾아야 하는지"를 정하는 일이 더 오래 걸릴 때가 많다.

특히 LLM에게 코드를 읽히는 워크플로우에서는 이 문제가 더 선명하게 드러난다. 함수 하나를 이해하는 일은 어렵지 않다. 하지만 다음과 같은 질문은 쉽게 느려진다.

- 주문 승인 같은 end-to-end 흐름은 어느 서비스들을 거치는가
- 여러 서비스가 공유하는 인프라는 어디에 묶여 있는가
- 설정과 DI 구성이 여러 파일에 흩어져 있을 때 전체 구조는 어떻게 생겼는가

이런 질문은 보통 `grep`과 파일 열람을 반복하면서 답하게 된다. 답은 얻을 수 있지만, 탐색 비용이 높다. 그래서 코드 인덱싱 기반 Knowledge Graph를 붙이면 이 비용을 줄일 수 있을지 실험해 봤다.

결론부터 말하면, **Knowledge Graph는 모든 질문을 더 빠르게 만들지 않았다. 하지만 특정 질문에서는 분명히 이겼다. 그리고 실전 ROI는 그래프의 품질보다 "어떤 질문을 그래프로 보낼지"를 정하는 라우팅 규칙에서 나왔다.**

## 왜 기존 탐색은 특정 질문에서 느려질까

단일 파일에 답이 있는 질문은 전통적인 코드 탐색이 여전히 빠르다.

- 특정 메서드 구현 상세를 묻는 질문
- 테스트 시나리오 목록처럼 한 파일 안에 답이 있는 질문
- 최신 변경 이유처럼 변경 이력이 더 중요한 질문

이 경우에는 파일 하나를 찾고 읽으면 끝난다. 그래프를 거치는 순간 오히려 단계가 하나 늘어난다.

반대로 아래 질문들은 구조적으로 느리다.

- 여러 서비스와 컴포넌트가 연결된 end-to-end 흐름
- cross-service shared infrastructure 식별
- 여러 Config 클래스에 흩어진 Bean 또는 설정 구조 파악

이때 느린 이유는 단순하다. 답이 한 파일에 없기 때문이다. 탐색자는 여러 파일을 순차적으로 열고, 각 파일이 가리키는 다음 파일로 계속 이동해야 한다. LLM도 똑같이 이 비용을 지불한다.

## 내가 실험한 방식

실험은 복잡하지 않았다. 두 가지 접근을 비교했다.

1. 직접 탐색
2. Knowledge Graph 기반 탐색

직접 탐색은 `grep`, `glob`, 파일 읽기만으로 답을 찾는 방식이다. Knowledge Graph 기반 탐색은 먼저 질문과 관련된 subgraph를 뽑아 보고, 필요한 경우에만 실제 코드를 다시 열어 확인하는 방식이다.

평가 포인트는 세 가지였다.

- 정확도가 유지되는가
- tool call이 줄어드는가
- 입력 토큰이 줄어드는가

질문은 대략 다음 범주로 나눴다.

- architecture
- function
- dependency
- config
- history

중요했던 점은 평균값만 보는 것이 아니라, **질문 유형별 편차**를 따로 본 것이다. 이 실험의 핵심 인사이트는 바로 여기서 나왔다.

## 결과: 평균은 비슷했고, 질문별 편차는 컸다

먼저 기대와 달랐던 점부터 말하자면, 전체 평균만 보면 두 방식은 거의 비슷했다. 정확도는 유지됐지만, 모든 질문에 그래프를 붙인다고 해서 자동으로 효율이 좋아지지는 않았다.

하지만 질문별로 나눠 보면 그림이 완전히 달라졌다.

### 그래프가 유효했던 질문

다음 유형에서는 그래프가 분명히 유리했다.

- end-to-end architecture 흐름
- 여러 서비스에 흩어진 shared infrastructure 식별
- 여러 Config 파일에 분산된 설정 구조 파악

이 범주의 일부 질문에서는 tool call이 30%에서 60% 이상 줄었다. 이유는 단순하다. 그래프가 답을 직접 만들어 준 것이 아니라, **탐색 범위를 먼저 압축해 줬기 때문**이다.

예를 들어 end-to-end 흐름을 설명하는 질문에서는, 직접 탐색 방식은 publisher, client, config, adapter를 하나씩 따라가며 파일을 여럿 열어야 했다. 반면 그래프 기반 접근은 관련 노드와 관계를 먼저 좁혀 준 뒤, 검증이 필요한 코드 몇 군데만 다시 읽으면 됐다.

cross-service 질문도 비슷했다. 직접 탐색은 서비스별 검색 결과를 손으로 합쳐야 하지만, 그래프는 애초에 연결 구조를 기준으로 정보를 뽑아낼 수 있다. 이 차이가 꽤 컸다.

### 그래프가 비효율적이었던 질문

반대로 아래 질문에서는 그래프가 오히려 오버헤드였다.

- 특정 메서드 구현 상세
- 테스트 메서드 또는 enum 목록처럼 단일 파일에 답이 있는 질문
- "왜 이렇게 바뀌었는가"처럼 코드보다 change history가 중요한 질문

이 경우에는 subgraph를 뽑는 단계 자체가 불필요했다. 이미 답이 한 파일에 있기 때문이다. 이런 질문은 차라리 정확한 파일을 바로 찾고 읽는 편이 빠르다.

## 핵심 교훈: 그래프의 ROI는 질문 라우팅에서 나온다

이 실험에서 가장 중요했던 결론은 이거였다.

> Knowledge Graph의 가치는 "항상 쓰면 빨라진다"가 아니라, "특정 질문에만 선택적으로 쓸 때 이득이 커진다"는 데 있다.

즉, 그래프를 도입하는 것 자체보다 먼저 해야 할 일은 **질문 라우팅 규칙**을 만드는 것이다.

실전에서는 대략 다음 기준이 유용했다.

### 그래프로 먼저 보낼 질문

- 여러 서비스나 컴포넌트를 가로지르는 흐름 질문
- "A, B, C가 공유하는 X는 무엇인가" 같은 shared infrastructure 질문
- 설정 또는 DI 구조가 여러 파일에 흩어져 있는 질문
- 두 개념 사이의 관계나 경로를 찾는 질문

### 직접 탐색으로 먼저 갈 질문

- 특정 함수나 메서드의 내부 로직
- 단일 파일 안의 열거형 목록
- 최신 변경 이력이나 설계 논의가 답에 더 중요한 질문

이 규칙 하나만 있어도 그래프를 무차별적으로 호출하는 실수를 줄일 수 있다. 그리고 그 순간부터 그래프는 "멋진 시각화 결과물"이 아니라, 실제로 탐색 비용을 줄이는 도구가 된다.

## 실무에서 써보며 느낀 제한점

몇 가지 제한도 분명했다.

### 1. 그래프는 최신 맥락을 항상 알지 못한다

코드 구조를 잘 보여주는 것과 "왜 이렇게 바뀌었는가"를 설명하는 것은 다른 문제다. 후자는 여전히 변경 이력과 설계 문서가 더 중요하다.

### 2. 범위를 너무 넓히면 오히려 덜 유용해질 수 있다

그래프는 커질수록 무조건 좋아지지 않았다. 너무 넓은 범위를 한 번에 묶으면 커뮤니티나 서브그래프가 다시 "잡탕"처럼 느껴질 수 있다. 실전에서는 적당히 범위를 좁힌 그래프가 더 유용했다.

### 3. LLM은 그래프 원본보다 잘 정리된 진입점을 더 필요로 한다

사람은 긴 wiki를 읽으며 스스로 맥락을 정리할 수 있다. 하지만 LLM은 그렇지 않다. 관련 없는 설명까지 한꺼번에 읽히면 오히려 비효율이 커진다. 그래서 사람이 보기 좋은 문서와 LLM이 쓰기 좋은 진입점은 다를 수 있다.

## 적용해 볼 만한 최소 규칙

지금 비슷한 시도를 하려는 팀이라면, 처음부터 큰 시스템을 만들기보다 아래 두 줄만 먼저 도입해 보는 것을 권한다.

1. cross-service, architecture, distributed config 질문은 그래프로 먼저 좁힌다.
2. function-level, single-file, change-history 질문은 직접 탐색으로 시작한다.

이 정도만 해도 실험의 절반은 끝난다. 나머지 절반은 그래프를 얼마나 정교하게 구축했는가가 아니라, 이 규칙을 얼마나 일관되게 운영하느냐에 달려 있다.

## 마무리

Knowledge Graph는 코드 탐색의 만능 해법이 아니었다. 하지만 그 점이 오히려 더 흥미로웠다. 정말 중요한 것은 "그래프를 썼는가"가 아니라 "언제 써야 하는가"였다.

LLM을 코드 탐색에 본격적으로 활용할수록, 검색 품질 못지않게 **질문을 올바른 탐색 경로로 라우팅하는 설계**가 중요해진다. 내게 이번 실험의 가장 큰 수확은 그래프 자체보다 그 운영 규칙이었다.

비슷한 실험을 한다면, 그래프 구축부터 시작하지 않아도 된다. 먼저 팀의 질문 유형부터 정리해 보자. 어떤 질문이 가장 자주 느린지 분류하는 순간, 그래프가 필요한 자리와 필요 없는 자리가 생각보다 빨리 드러난다.
